# 최종 판정

**실패 — keyman_drop (1라운드 종결).**

개인 소비자(`decision_authority: full`, `stakeholders: []`) 구성이라 keyman인 km 본인이 곧 모든 의사결정자다. 01 세션에서 `decision: drop`, `confidence: 55`로 초기 판정되었고, 성사 조건인 `decision == convince_stakeholders AND confidence > 75`를 둘 다 충족하지 못한다. stakeholder가 없으므로 5b~5c 루프 자체가 불필요하게 되어 후속 세션(02~05)은 생성되지 않았다.

drop 사유의 핵심은 세 가지다:
1. 핵심 가치(자동 rep 카운트 정확도)의 정량 근거 부재 — 프로파일의 1순위 reject_trigger("rep 카운트가 자주 틀려서 신뢰 붕괴")에 직격.
2. 외부 신뢰 신호(앱스토어 별점·리뷰) 0 — 프로파일의 communication_style("리뷰·앱스토어 별점으로 의사결정")과 정면 충돌.
3. 실질 커버리지가 스쿼트 단일 운동 — 사용자의 루틴(스쿼트/푸쉬업/플랭크/런지)의 1/4만 대체, 병행 툴화 → 설치 동기 약화.

# 실행 리스크

**낮음 (가정적).** `decision_authority: full`이고 직접/간접 stakeholder가 없는 개인 소비자 페르소나이므로, 만약 km이 수용(accept)했다면 도입과 사용 사이의 조직적 마찰은 사실상 0이다. 다만 시뮬 상으로는 애초에 통과하지 못했으므로 실제 도입 자체가 발생하지 않는다.

남는 리스크 관점 — 본인 결정 후에도 "3일 써보고 아니면 삭제" 룰과 "설치 후 첫 10분에 막히면 3분 컷" 패턴이 프로파일에 강하게 박혀 있어, 설령 설치까지 갔어도 첫 세션에서 손바닥 인식 실패나 rep 오카운트가 1회만 발생해도 즉시 삭제로 직결될 수 있다. 즉 "도입 리스크"는 낮지만 "잔존 리스크(retention risk)"는 여전히 높다.

# 가치제안 개선 포인트

1. **정량 정확도 공개 (rep 카운트 %)** — 1건(01_keyman_initial). 대표 발화: "내 reject_trigger 1순위가 'rep 카운트가 자주 틀려서 신뢰 붕괴'인데, 가치제안 §1·§5는 '내부 개발 중 관측한 감각으로는 대체로 안정적' 같은 주관 표현뿐이고 % 수치는 '베타 이후 측정해 갱신 계획'이라고 미룬다." → 스쿼트라도 먼저 내부 샘플 기준 정확도(예: N=50 세션, 오카운트율 X%)를 숫자로 제시 필요.
2. **실패 복구 경로 (손바닥 폴백 + 에러 가이드)** — 1건. 대표 발화: "손바닥이 1초 인식 안 되면 카운트다운이 아예 시작되지 않고 화면 탭 같은 대체 트리거가 없다." + "왜 안 되는지 안내가 없는 상태에서 불안정한 인식을 겪으면 원인을 혼자 못 찾아내고 바로 삭제로 간다." → 화면 탭 대체 트리거 + "왜 인식 안 되는지" 실패 원인 오버레이 UI 추가.
3. **외부 신뢰 신호 부재 (앱스토어 출시·리뷰·별점)** — 1건. 대표 발화: "나는 리뷰·별점으로 앱을 결정하는 사람이다. 별점·리뷰가 하나도 없는 상태에서 '이 문서 자체의 투명 고지'를 신뢰 신호로 내세우는 건, 나한테는 신뢰 신호가 아니라 '아직 검증이 없다는 사실의 재확인'에 가깝다." → 앱스토어 출시 타임라인 명시, 100명 베타 피드백 공개, 최소한 베타 배포 경로(TestFlight 링크)라도 value_prop에 포함.
4. **커버리지(스쿼트 단일 운동) 문제** — 1건. 대표 발화: "내가 집에서 돌리는 루틴(스쿼트/푸쉬업/플랭크/런지)의 1/4밖에 안 커버된다. 그럼 나는 여전히 유튜브 홈트 + 수동 카운트를 병행해야 하고, Mofit은 추가로 얹는 툴이 된다 — 설치 동기가 약해진다." → 푸쉬업·플랭크 중 최소 1종이라도 실제 카운팅까지 되는 상태로 확장하거나, "스쿼트 전용 카운터"로 포지셔닝을 좁혀 기대 수준 정렬.
5. **로컬 기록 로그인 후 미이관 (ADR-013)** — 1건. 대표 발화: "3일 써서 쌓인 기록이 로그인 순간 통째로 리셋되는 설계를 만나면 기분 상한다. 이 설계를 의도라고 우기는 게 오히려 불신 신호다." → 설계 자체를 재검토하거나, 최소 "로그인 시점에 로컬 기록 서버 병합" 옵션을 UX에 노출.

7. **권장 환경 외 실사용 데이터 부재** — 1건. 대표 발화: "권장 환경(밝은 실내·2~3m·전신)을 못 맞추는 경우(좁은 원룸, 야간 운동) 카운트가 얼마나 튀는지에 대한 실제 데이터가 전혀 없다." → 좁은 공간/저조도 조건에서의 degradation 수치 또는 해당 환경 감지 시 경고 UI 필요.

# 페르소나 보정 힌트

- **파일: 01_keyman_initial.md — 일관성 양호, 부자연스러운 어긋남 없음.** 프로파일의 `reject_triggers[2]`("rep 카운트가 자주 틀림 → 신뢰도 붕괴"), `communication_style`("리뷰·앱스토어 별점으로 의사결정"), `risk_preference: conservative`, `trust_with_salesman: 40` 값과 drop 사유가 일관적으로 얼라인된다. 특히 "별점·리뷰가 0인데 문서의 투명 고지를 신뢰 신호로 내세우는 건 신뢰 신호가 아니다"라는 발화는 프로파일을 그대로 활용한 판단이다.
- **파일: 01_keyman_initial.md — unknown 필드 기반 비관 편향은 있으나 페르소나와 정합적.** rep 정확도 수치가 value_prop에 비어 있는 것을 "측정 안 됨"으로 해석하고 감점한 부분이 있는데, 이는 `risk_preference: conservative` + `tech_literacy: medium`(조잡하면 즉시 이탈) 속성과 정합적이므로 보정 불요. 향후 캘리브레이션에서 "증거 부재 → 감점" 강도를 조금 완화할지 여부는 다른 페르소나 결과와 교차해서 판단 권장.
- **파일: 01_keyman_initial.md — confidence=55는 "drop이지만 전면 거부는 아님" 신호.** km이 "나중에 앱스토어에 실제로 올라오면 다시 본다" 식 유보 포지션을 명시했는데, 이는 프로파일의 `buy_triggers`(촬영 없이 자동 카운팅·로그인 없이 시작 가능)를 원론적으로는 긍정하고 있다는 증거. "rep 정확도 수치 + 앱스토어 출시" 두 축만 보강되면 재평가 시 accept 쪽으로 이동할 여지가 있음을 시사하므로, 이 페르소나는 "1회 drop = 영구 탈락"으로 취급하지 않는 편이 정확.
